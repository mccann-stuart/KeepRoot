## 2025-05-18 - SSRF in Scheduled Source Syncing
**Vulnerability:** The RSS/Atom source syncing feature directly fetched an external URL (`source.pollUrl`) constructed from a user-provided `bridgeUrl` without any loopback/SSRF protection or safe redirect handling, exposing internal network services to Server-Side Request Forgery via scheduled Worker tasks.
**Learning:** Scheduled or background tasks that consume user-configured URLs must be subjected to the exact same SSRF mitigations as immediate, user-facing URL ingests.
**Prevention:** Always run user-supplied hostnames through a DNS-based blocklist filter (`validateSafeUrl`) and configure `fetch` with `redirect: 'manual'`, strictly validating every redirect `Location` header before following. Ensure any `Response` variable from a redirect loop handles null checks properly to satisfy TypeScript compilation.

## 2024-04-27 - SSRF Bypass via Fetch Redirects
**Vulnerability:** Server-Side Request Forgery (SSRF) bypass due to `fetch()` automatically following redirects after an initial `validateSafeUrl` check. An attacker could provide a safe URL that redirects to a malicious internal/private IP (e.g., `http://127.0.0.1`), which `fetch()` would blindly follow.
**Learning:** Initial validation of a URL is insufficient if the HTTP client automatically follows redirects to unvalidated destinations.
**Prevention:** Always configure `fetch()` with `redirect: 'manual'` when handling user-provided URLs. Implement a manual redirect loop (e.g., up to 5 redirects) that explicitly extracts the `Location` header, parses it against the current URL context, and validates the new URL using `validateSafeUrl` before continuing the fetch. Also ensure to safely cancel bodies of 3xx responses (e.g., `await response.body?.cancel().catch(() => {})`) to avoid memory leaks or DoS.
