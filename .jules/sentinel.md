## 2024-05-20 - [Denial of Service via Uncaught Invalid URL Exception in Redirects]
**Vulnerability:** The application was vulnerable to crashing or throwing uncaught exceptions when parsing malformed or invalid `Location` headers during manual redirect following. The unhandled `new URL(location, currentUrl)` exception could disrupt ingestion pipelines.
**Learning:** This existed because the assumption was made that standard HTTP clients or remote servers would always provide well-formed URL headers, leading to a lack of defensive `try...catch` blocks around the native `URL` constructor.
**Prevention:** Always wrap `new URL()` instantiation in `try...catch` blocks when parsing untrusted input, including headers received from third-party servers, to fail gracefully rather than crashing the execution context.
## 2025-02-24 - SSRF Bypass via Embedded Credentials
**Vulnerability:** The application's `validateSafeUrl` hostname validation to prevent SSRF could be bypassed by exploiting URL parsing logic discrepancies, specifically by embedding credentials in the URL structure like `http://user:pass@127.0.0.1`.
**Learning:** URL parsers in different execution environments (Node.js/Workers vs. underlying native `fetch` client) may disagree on whether the `127.0.0.1` part is the hostname or part of the credentials when `google.com` is suffixed (e.g., `http://127.0.0.1@google.com`). This mismatch allows attackers to spoof validation checks while tricking the HTTP client into routing locally.
**Prevention:** Always explicitly check for and reject embedded credentials (`parsedUrl.username` and `parsedUrl.password`) within the URL object before proceeding with any hostname-based SSRF blocklist validation.
