## 2024-05-24 - Prevent SSRF Bypasses via URL Parser Discrepancies
**Vulnerability:** The `validateSafeUrl` function used the native `URL` constructor to validate hostnames against an allow/blocklist, but failed to restrict embedded credentials. This allows attackers to craft URLs with malformed authority strings (e.g., `http://1.2.3.4\@example.com`) that bypass hostname validation (the parser thinks the host is `example.com`) but can trick the underlying `fetch` implementation into sending the request to the IP/hostname before the `@` symbol.
**Learning:** URL parser differentials between JS `new URL()` and the underlying network stack are a common vector for SSRF bypasses. The native `URL` constructor is not a strict validator and will often cleanly parse malformed URLs, hiding the credentials or failing to extract them into `.username` and `.password` properties, leaving them in the `.hostname` or `.pathname`.
**Prevention:** Always explicitly reject URLs that contain basic authentication credentials (`.username` or `.password`). Further, manually verify the raw string authority section (everything before the first `/`, `?`, or `#`) to ensure it does not contain the `@` symbol, effectively preventing any credential-based URL parser tricks.
