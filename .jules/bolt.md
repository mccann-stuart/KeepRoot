## 2024-05-18 - Parallelizing D1 and R2 Reads
**Learning:** In Cloudflare Workers, retrieving a complex entity like a bookmark often requires querying D1 for metadata and R2 for object storage content. Previously, `getBookmark` executed a sequential chain: fetch `bookmarkRow` -> fetch `contentRow` -> fetch `[contentDocument, tags, images]` via `Promise.all`. However, because the queries for `contentRow`, `tags`, and `images` all only depend on `bookmarkId`, they can be batched together into a single `D1Database.batch()` call. This eliminates two sequential HTTP network roundtrips to the D1 database. Furthermore, the R2 fetches (`getContentDocument` and `htmlObject.text()`) can then be parallelized via `Promise.all()` directly after the combined D1 batch returns.
**Action:** Always identify independent database reads that depend on the same parent ID (like `bookmarkId`) and group them into a single `D1Database.batch()` call before firing parallel object storage fetches.
## $(date +%Y-%m-%d) - Preserve Ordering When Parallelizing Network Requests
**Learning:** When refactoring sequential loops that populate an array into concurrent `Promise.all()` executions (like fetching multiple images), mutating the shared array inside the async callbacks using `.push()` destroys the original ordering because it appends results as they complete. In contexts where order is critical (e.g., preserving the prioritized `og:image` as the first thumbnail), this creates a functional regression.
**Action:** When parallelizing ordered processing, always map to an array of promises, `await Promise.all()` on that array, and then process the results sequentially to build the final array, preserving the initial order.
## 2025-04-28 - Avoid Using .push() to Optimize .map()
**Learning:** JS engines like V8 optimize `.map()` by pre-allocating an array of the exact known size. Blindly replacing `.map()` with a procedural loop that uses `.push()` on an empty array forces dynamic reallocation and increases overhead, creating a micro-optimization with negative impact.
**Action:** Only replace `.map()` with a procedural loop when it eliminates the need to allocate an intermediate array entirely, such as populating a `Map` or directly aggregating results.
## 2025-05-04 - O(N) Database Reads for Search Queries
**Learning:** In Cloudflare Workers, retrieving records to apply in-memory filtering (like search results) by fetching all of a user's records (O(N) where N is total bookmarks) causes severe memory spikes and latency, particularly as user data grows. This can easily lead to OOM errors in memory-constrained environments (like Workers' 128MB limit). Pushing the filtering down to the database level using `IN (...)` clauses allows for retrieving only the necessary subset. However, because Cloudflare D1 does not support binding arrays directly to `IN` clauses, the array must be chunked (e.g., 50 items) to respect SQLite parameter limits, and the placeholders must be dynamically generated. Since SQLite does not guarantee order when concatenating results from multiple chunked queries, the results must be manually re-sorted in JavaScript to maintain the application's required ordering.
**Action:** Always avoid O(N) full-table reads into memory for subset filtering. Use chunked `IN (...)` clauses with dynamic placeholders to fetch specific IDs from D1, and manually sort the concatenated chunk results in memory.
## 2025-05-05 - Concurrent execution of FTS and Vector Search
**Learning:** FTS (Full Text Search) and Vector search operations are independent of each other when computing search scores. Running them concurrently with \`Promise.all()\` instead of sequentially reduces overall search latency by parallelizing database and AI model queries.
**Action:** Identify independent operations, especially network or AI model calls, and execute them concurrently using \`Promise.all()\` whenever possible to reduce overall latency.
